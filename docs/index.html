Bring your own AI agent

The official way to set up a ZGX nano is a VS Code extension. That is one editor, one workflow, one box you have to sit in front of. ZGX CLI is just commands — so anything that can run a command can drive your device, including your AI coding agent.

  1. Install the CLI. One binary, every OS (see below).
  2. Add the agent skill. One file. It teaches your agent the exact commands, the gotchas, and self-checking recipes that were verified against real hardware.
  3. Ask in plain language. Your agent discovers and verifies on its own, and hands you the exact command whenever a password is needed.
you ▸ get my ZGX nano ready for development agent ▸ discovering on the LAN… found spark-7f2a agent ▸ run this — it'll ask for the device password: zgx connect spark-7f2a --user demo you ▸ ·········· → Key-based access works. agent ▸ zgx health spark-7f2a --user demo # → healthy agent ▸ now run this — it'll ask for your sudo password: zgx install jupyter-lab podman --host spark-7f2a --user demo you ▸ ·········· → Installed agent ▸ zgx verify jupyter-lab podman --host spark-7f2a --user demo # → ✓ all installed agent ▸ done. What it actually does

Everything the desktop tooling does, as plain commands you (or your agent) can script, pipe and verify.

  • Finds the device for you. Zero-config mDNS discovery on the LAN — no IP hunting.
  • Sets up passwordless SSH safely. ed25519 keys, trust-on-first-use host-key confirmation, idempotent — re-run it without fear.
  • Installs a curated AI/ML stack. 17 apps (Jupyter, Ollama, podman, miniforge, and more) in dependency order, with clear per-app success/failure reporting.
  • Shows its work. Prints the raw remote commands and flags any curl | sh before it runs — no surprise code on your box.
  • Pairs two units over ConnectX. Discovers ConnectX NICs and writes link-local netplan for device-to-device networking (implemented; two-device verification still pending).
  • One binary, every desktop. Pure-Go, no cgo — Linux / macOS / Windows, amd64 + arm64. Nothing to install on the device itself.

Install

Pick whichever is easiest — the binary lives on your laptop, not the device.

# macOS / Linux — Homebrew (auto-updates):
brew install --cask AndersSol/tap/zgx

# macOS — double-click installer:
#   download the .pkg above (signed + notarized — no Gatekeeper warning), double-click, done.

# Any OS — with Go 1.26+ (this builds a binary named `zgx-cli`; rename to `zgx` if you like):
go install github.com/AndersSol/zgx-cli@latest

# …or grab a binary from the download buttons above and put it on your PATH.

Quick start (from the terminal)

zgx discover                                          # find devices on the network
zgx connect spark-7f2a --user demo                    # set up SSH key access (type the device password)
zgx list                                              # browse the curated app catalog
zgx install jupyter-lab --host spark-7f2a --user demo # install (dependencies pulled in automatically)
zgx verify  jupyter-lab --host spark-7f2a --user demo # confirm it landed
zgx health  spark-7f2a --user demo                    # check connectivity any time

First device? The default SSH user is hp on HP factory images, but a re-imaged DGX Spark uses the name you created at first boot — pass --user <name>. The CLI tells you when that's the problem.

Safe by default

It configures a machine over SSH, so it treats that like it matters.

  • Host-key TOFU + fingerprint. Confirms an unknown host's fingerprint before the password is sent; rejects changed keys.
  • No surprise code. Raw remote commands shown; download-and-execute lines require explicit confirmation.
  • Input validation. No SSH-config injection; NIC names and device IDs are validated before use in remote commands.
  • No secrets. Passwords are never logged or written to disk; the repo is gitleaks-clean.
